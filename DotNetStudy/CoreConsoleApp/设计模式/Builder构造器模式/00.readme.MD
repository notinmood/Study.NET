# 说明

## 使用场景
复杂对象的创建。
- 场景1.这个复杂对象有很多个步骤，并且这些步骤的前后顺序流程是确定的，所不确定是每个步骤的实现方式。
	- 生成器模式的重心在于分离构建流程的算法和具体的构造实现，从而使得构建算法可以重用，具体的构造实现可以很方便的扩展和切换。
	- 即：将一个复杂对象的构建流程与它的构建细节分离，使得同样的构建过程可以创建不同的构建表示。
- 场景2.这个复杂对象有多个属性需要设置，因此有些时候容易遗漏。这样前面进行各种准备（对各个属性进行设置），最后通过Build得到最后的结果。



## 构造者模式vs抽象工厂
都是创建复杂对象。
- 抽象工厂创建的复杂对象内部各个部件，有关联，但没有流程顺序。
- 构造者模式创建的复杂对象内部各个部件，有前后的流程顺序。更关注的是这个流程过程。

## 构造者模式vs模板方法模式
两者都包含有对流程的控制，在有些情形下，两者可以互相替代。但这取决于具体的需求和上下文。如果主要关注算法的不同实现，模板方法模式可能更合适。如果关注于对象的构建过程和灵活性，建造者模式可能更合适。
- 构造者模式是创建型模式，目的是创造产品。（当然在创造产品的过程中需要按照流程进行，但其更关注最后的产品）
- 模板方法模式是行为型模式，其关注的重点的流程过程。



## 示例说明
这是一个将业务数据从系统导出到其他地方的示例。

- 在没有使用设计模式前，所有的逻辑都写在每个Exporter里面（包括构建流程和构建每个部分的细节），这样流程过程在多个Exporter间是重复的。如果以后流程修改的话，要在多处进行修改。这违反了DIY原则。
- 使用Builder模式后，将流程算法单独提取出来。以后流程有变，就只修改流程所在的Director文件就可以了。
